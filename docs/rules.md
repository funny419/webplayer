# 팀 운영 규칙 (Rules)

> 최종 업데이트: 2026-04-20 | 관리: PM

---

## 1. 완료 판단 기준

- **GitHub Issue Close**를 완료 기준으로 한다.
- 프로젝트 보드 Status(Done)와 Issue Close를 **반드시 동시에** 처리한다.
- 프로젝트 보드 Done 상태만으로는 완료로 인정하지 않는다.

## 2. 의사결정 및 승인

- 마일스톤 전환, 팀원 새 작업 지시, 우선순위 변경 등 **주요 결정 전 반드시 팀장(사용자)에게 먼저 보고**하고 승인을 받는다.
- 팀원이 완료 보고를 해도 **팀장 승인 없이 다음 단계를 자동으로 시작하지 않는다**.
- 승인 흐름: `팀원 완료 보고 → PM 집계 → 팀장 보고 → 승인 → 다음 단계 지시`

## 3. 블로커 보고

- 블로커(의존성 미완료, 기술적 막힘 등) 발생 시 **즉시 PM에게 보고**한다.
- PM은 블로커를 확인 후 팀장에게 보고하고 조율한다.

## 4. 담당자 배정 기준

- 이슈 레이블 기준으로 담당자를 배정한다.

| 레이블 | 담당자 |
|--------|--------|
| `web`, `gameplay` | web-dev |
| `android` | android-dev |
| `design` | designer |
| `planning` | game-planner |

- 복합 레이블 이슈(예: `web` + `design`)는 PM이 협업 방식을 정의한다.

## 5. 개발 우선순위

- **웹 게임 완성 우선**: M1~M6(웹 게임)을 완전히 완료한 후 M7(Android 배포)을 진행한다.
- android-dev는 웹 게임 완성 전까지 대기하며, **팀장 지시 시에만 작업을 재개**한다.

## 6. 마일스톤 관리

- 마일스톤의 모든 이슈가 Close될 때까지 완료를 선언하지 않는다.
- PM은 마일스톤 잔여 이슈를 주기적으로 GitHub에서 직접 확인한다 (팀원 보고에만 의존하지 않는다).
- 마일스톤 완료 시 팀장에게 보고하고, 다음 마일스톤 담당자 배정 계획을 승인받는다.

## 7. 커뮤니케이션

- 팀원 간 주요 결정은 SendMessage로 기록을 남긴다.
- PM은 팀장의 지시를 전체 팀에 공지할 의무가 있다.
