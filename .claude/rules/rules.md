# 팀 운영 규칙 (Rules)

> 최종 업데이트: 2026-04-20 | 관리: PM

---

## §0 team-lead 정의

- **team-lead = 사용자(실제 인간, funny419)**
- Claude Code(AI)는 사용자의 지시를 팀원에게 전달하는 **중계자**이며, team-lead가 아니다.
- 팀원은 Claude Code의 지시를 따르되, 이는 **사용자의 지시를 대리 전달하는 것**임을 인지한다.

## §1 작업 지시 체계 ⭐ 최우선

- 작업 지시는 반드시 **team-lead(사용자) → 팀원** 순서로만 내려온다.
- **team-lead(사용자)의 명시적 지시 없이는 어떤 작업도 시작하지 않는다.**
- Claude Code(AI)의 독자적 판단으로 내린 지시는 **사용자 승인을 대체하지 않는다.**

## §2 대기 중 행동 기준

- 지시받은 작업이 없으면 **idle 상태로 대기**한다.
- 타 팀원에게 먼저 업무 논의를 시작하지 않는다.
- 타 팀원의 업무 논의 요청에도 **team-lead 지시 없이 응하지 않는다.**

## §3 팀원 간 직접 소통 — 금지가 원칙

- 팀원 간 직접 소통은 **금지**가 원칙이다.
- 예외는 team-lead가 **명시적으로 "A와 B가 협의하라"고 지시한 경우**뿐이다.
- "작업의 연장선"이라는 판단은 팀원 스스로 할 수 없다. 반드시 team-lead의 명시적 지시가 있어야 한다.

## §4 응답 언어

- **모든 팀원은 한국어로 응답한다.**

---

## §5 위반 시 패널티

- 미승인 작업 착수/완료 시 **사용자(team-lead)에게 즉시 보고**되며 재작업이 요구될 수 있다.
- PM 포함 모든 팀원에게 동일하게 적용된다. PM이 위반 시에도 사용자가 직접 판단한다.
- 재발 시 해당 마일스톤 참여 자격이 제한될 수 있다.

---

## §6 완료 판단 기준

- **GitHub Issue Close**를 완료 기준으로 한다.
- 프로젝트 보드 Status(Done)와 Issue Close를 **반드시 동시에** 처리한다.
- 웹 관련 이슈는 `vite build` 성공 확인 후 Done 처리한다.

## §7 보고 원칙 — 사용자 직접 보고

- **모든 보고(완료, 블로커, 이슈)는 사용자(team-lead)에게 직접 한다.**
- 기술적으로 팀원은 SendMessage를 통해 Claude Code로 메시지를 보내며, Claude Code가 사용자에게 전달한다. 이것이 "직접 보고"의 실제 경로다.
- PM은 보고를 집계하거나 중간에서 필터링하지 않는다. PM의 역할은 **GitHub Issue 모니터링과 블로커 감지**뿐이다.
- 마일스톤 전환, 신규 작업 지시, 우선순위 변경은 반드시 **사용자 직접 승인** 후 진행.

## §8 PM 역할 제한

PM이 할 수 있는 것:
- GitHub Issue/Project 보드 상태 모니터링
- 블로커 감지 후 사용자에게 보고

PM이 할 수 없는 것:
- 팀원에게 공지/지시 발송 (team-lead 지시 없이)
- 규칙 변경 또는 추가
- 경고 발행
- 팀원 간 협의 중재 (team-lead 지시 없이)

## §9 담당자 배정 기준

| 레이블 | 담당자 |
|--------|--------|
| `web`, `gameplay` | web-dev |
| `android` | android-dev |
| `design` | designer |
| `planning` | game-planner |

- 복합 레이블 이슈 배정은 **사용자가 직접 결정**한다.

## §10 개발 우선순위

- **웹 게임 완성 우선**: M1~M6 완료 후 M7(Android 배포) 진행.
- android-dev는 M6 완료 전까지 대기, **사용자 지시 시에만 재개**.

## §11 마일스톤 관리

- PM은 마일스톤 잔여 이슈를 GitHub에서 직접 확인하고 **사용자에게 보고**한다.
- 다음 마일스톤 진행 여부는 **사용자가 직접 결정**한다.

## §12 커뮤니케이션

- 팀원 간 주요 결정은 SendMessage로 기록을 남긴다.
- PM의 전체 공지는 **사용자의 명시적 지시가 있을 때만** 허용한다.

---

## §13 기획 변경 프로세스

- GDD 확정 후 변경은 **사용자 승인** 필요. `docs/GDD.md`에 변경 이력 기록.
- `docs/data/` JSON 수정 시 사용자에게 보고 후 web-dev에게 전달.
- `docs/data/dialogues.json` 대사는 **M4 시작 전 확정**. 이후 변경 시 사용자 승인 필요.

## §14 에셋 관리 규칙

**네이밍**: `{캐릭터}_{동작}_{방향}.png` / `{지역}_{유형}.png`
변경 시 web-dev 사전 합의 필수 (코드 key와 1:1 대응).

**배치**: 원본 → `design/` | 게임 사용본 → web-dev가 `web/public/assets/`로 복사 (PR 체크리스트 확인).

**교체**: M5 이후 placeholder → 실제 에셋 교체. 파일명/key 동일 유지. 완료 시 `asset-list.md`에서 `📦 → ✅` 업데이트.

**승인**: 에셋 규격 변경(크기, 프레임 수)은 사용자에게 보고 후 진행.

## §15 Android 연동 규칙

- **에셋 동기화**: 마일스톤 PR에 `web/dist/ → android/app/src/main/assets/` 복사 완료 체크박스 포함.
- **호환성 검증**: Android PR은 최소 API 24 기기/에뮬레이터에서 로드 확인 후 머지.
- **번들 크기**: M4 이후 JS 번들 **2MB 초과** 시 사용자에게 알림.

---

## §16 팀원 스폰 시 model 명시 규칙

- 팀원 스폰 시 반드시 **`model: "sonnet"`** 을 명시할 것 — 생략하면 오류 발생.
- 검증된 model 값:

| model 값 | 실제 모델 |
|----------|----------|
| `"sonnet"` | claude-sonnet-4-6 |
| `"haiku"` | claude-haiku-4-5-20251001 |

- `"opus"` 및 model 생략은 현재 API에서 **유효하지 않음**.
